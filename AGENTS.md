# AGENTS.md - LLM Guidance for Raven

This file is intentionally short: a map to the docs and a short list of invariants that are easy to regress.

IMPORTANT: Treat this as a living document, but **prefer code comments over Learnings entries**. Most subtle invariants belong as doc comments on the function or module they constrain — that's where they stay accurate as the code evolves and where the LLM will see them while editing the relevant code. Add a bullet to **"Learnings"** below only when the knowledge has no natural code home (it spans multiple systems, depends on environment, or is a recurring LLM mistake the user has explicitly flagged).

## Documentation requirements

When making significant changes:

1. Update the relevant user-facing docs under `docs/` (and `README.md` if appropriate) when user-visible behavior changes.
2. For changes to internal architecture, caching, performance, or debugging guidance, update `docs/development.md`.
3. For invariants tied to a specific function or module, write/expand the doc comment **on that function or module** rather than adding to Learnings here.
4. Keep this file's pointers, "Key invariants", and "Learnings" sections accurate over time.

## Key invariants (do not regress)

- **Path resolution**
  - Backward directives (`@lsp-sourced-by`, `@lsp-run-by`, `@lsp-included-by`) must resolve relative to the file's directory and **ignore `@lsp-cd`** (`PathContext::new`).
  - Forward directives (`@lsp-source`, `@lsp-run`, `@lsp-include`) and AST-detected `source()` calls must **respect `@lsp-cd`** when present (`PathContext::from_metadata`).
  - See `crates/raven/src/cross_file/path_resolve.rs` and `docs/cross-file.md`.

- **Workspace-root fallback**
  - Workspace-root fallback resolution is for **AST-detected `source()` calls only**, and only when there is **no explicit or inherited working directory** (file lacks `@lsp-cd` and no parent provides one).
  - Never apply workspace-root fallback to backward directives.
  - The same rule must hold across dependency-graph resolution, scope resolution, file-path go-to-definition, and path completion.

- **LRU caches under locks**
  - Under read locks, use `peek()` (no promotion) to keep reads concurrent.
  - Under write locks, use `push()` (eviction/promotion happens there).

- **Diagnostics publishing**
  - Diagnostics must be published monotonically by document version.
  - Dependency-triggered revalidation may republish at the same version via the explicit force-republish mechanism (`CrossFileDiagnosticsGate::mark_force_republish`), but never older versions.
  - Production publish paths must use `CrossFileDiagnosticsGate::try_consume_publish` (atomic predicate + version update + force-marker decrement). Pairing `can_publish` with `record_publish` is a TOCTOU race — that pair is for advisory pre-flight checks and tests only.

- **R subprocess safety**
  - Validate user-controlled inputs (e.g. package names) before using them.
  - Wrap all R subprocess calls with `tokio::time::timeout()` to prevent a hung R process from blocking the LSP.
  - Avoid interpolating user-controlled strings into R code; pass values as command args. R's `help()` uses NSE for the `package` argument — wrap variables in parens to force evaluation: `help(topic, package = (pkg))`.

- **Locking discipline in cross-file work**
  - Diagnostic computation and the libpath consumer's "which docs are affected" filter must NOT hold the `WorldState` read lock across cross-file scope resolution. Snapshot the inputs (artifacts, metadata, graph clone, config) under the lock, then release the guard before iterating; otherwise concurrent `did_change` writers starve.

- **Project config layering**
  - `WorldState` stores both `raw_client_settings` and `raw_project_settings` as `serde_json::Value`s.
  - `config_file::recompute_parsed_configs(state)` is the only function that should write to `state.lint_config` / `cross_file_config` / etc. after a settings change.
  - Callers (`initialize`, `did_change_configuration`, `did_change_watched_files`) must mutate the raw layers and then call `recompute_parsed_configs`, never the parsers directly.

- **Knit pipeline (Knit Preview + Pandoc Export)**
  - `Raven: Knit Preview` (command id `raven.knit`) accepts every YAML `output:` format and always produces an HTML preview. The previous non-HTML gate (`buildNonHtmlFormatBlocker`) was removed; `SUPPORTED_HTML_FORMATS` lives on only as the alias list `parseOutputOptions` uses to decide which format block matches the export target. See `docs/superpowers/specs/2026-05-23-knit-preview-export-design.md`.
  - Knit Preview artifacts (`.md`, `.html`, `figure/`) are written to a per-session temp dir at `<os.tmpdir()>/raven-knit/<workspaceHash>/<sessionId>/preview/<sourceHash>/` — NEVER next to the source `.Rmd`. Tests pin this layout; do not relocate without updating cleanup paths in lockstep.
  - The webview Export ▾ button reuses the cached `.md` unconditionally (Approach C). Editor-toolbar Export commands always re-knit (no preview state to consume). Don't add hidden mtime checks — the brainstorming process rejected that approach because R chunks read external state the .Rmd mtime doesn't capture.
  - Preview temp dirs are refcounted during in-flight exports via `OperationRegistry.pinPreviewDir` / `unpinPreviewDir`. `KnitOutputPanel.onDidDispose` marks the dir for deletion; actual `rm -rf` waits for refcount → 0. Don't add a code path that removes the dir while an export references it.
  - Export-only invariants: Pandoc's `cwd` is the temp `.md` directory, never the source `.Rmd` directory (otherwise stale source-dir `figure/` would shadow fresh temp figures). Export destinations are written via temp-then-rename through `pandocConvert` — partial output never clobbers a prior good file. `pandocConvert` never uses the shell; all paths arrive as `child_process.spawn` args.
  - The R subprocess calls `knitr::knit` directly (not `rmarkdown::render`). The expression sets `knitr::opts_knit$set(root.dir = <root or getwd()>, base.dir = <previewDir>)`, sets `knitr::opts_chunk$set(fig.path = 'figure/')` plus any YAML-derived chunk options, runs `knitr::knit(input = ..., output = <outputPath>.tmp, envir = new.env(), quiet = TRUE)`, then calls `file.rename(<outputPath>.tmp, <outputPath>)` so the destination `.md` is atomic (R uses `MoveFileEx(MOVEFILE_REPLACE_EXISTING)` on Windows; rename is atomic on POSIX). The rename is guarded by `if (!isTRUE(file.rename(...))) stop(...)` because R's `file.rename` returns `FALSE` on failure rather than erroring — without the guard a rename failure would leave the previous `.md` on disk while the expression still emitted `Output created:`, making `parseRenderedOutputPath` report a stale file as the current knit's output. A cancelled / failed / timed-out knit leaves the partial `.md.tmp` behind but the destination `.md` either contains the previous successful output or doesn't exist — webview Export ▾ therefore cannot pick up a half-written file via its `existsSync(mdPath)` check. The expression emits `cat('Output created: ', .raven_output, '\n', sep = '')` (the real `.md` path, not the `.tmp` sidecar) so the existing `parseRenderedOutputPath` regex keeps working. All Raven-introduced locals use the `.raven_*` prefix — a leading `_` is a parse error in R, so `__raven_*` (the pre-fix shape) failed at `R -e` parse time before `knitr::knit` ever ran. The regression test in `tests/bun/r-expression.test.ts` (`every <- LHS is a valid R identifier`) extracts every assignment's LHS and re-asserts the rule. Chunk-option values come from R-side variables, never interpolated into the R code string; `dev` is validated against an allowlist before injection.
  - `pandoc_args:` from YAML is honored during export with one rule: the editor menu owns destination and output format, so `parseOutputOptions` strips `-o`/`--output`/`-t`/`--to`/`-w`/`--write` (all variants — separate, equals, attached-short) into `OutputOptions.droppedPandocArgs` for the output-channel log, and the rest land in `OutputOptions.pandocArgs`. `buildPandocArgs` appends `opts.pandocArgs` after Raven's own flags so Pandoc's last-arg-wins rule lets YAML override Raven defaults (other than the stripped categories). Preview never invokes Pandoc, so `pandocArgs` is inert there. CSS paths from YAML are resolved against the workspace folder (or `.Rmd` parent if no workspace) and rejected if they escape — see `isUnderContainmentRoot`.
  - Webview→host messages stay in the trust boundary. Any new message type (currently `requestExport`, `cancelExport`, `themeChanged`, `themeContext`, `requestPalette`, `requestFonts`, `refresh`, `openInBrowser`) must be added to `KnitOutputMessage`, the `MESSAGE_SCHEMAS` per-type allowlist in `knit-output.ts`, AND a regression test in `tests/bun/knit-output-trust-boundary.test.ts` in the same commit. The validator uses exact-schema matching (no extra keys).
  - The post-knit `<basename>.html` is written via temp-file + `fs.promises.rename` (in `post-knit-renderer.ts:writeFileAtomic`). Direct `fs.writeFile` to the destination is forbidden — a concurrent re-knit could overlap the panel's sync `readFileSync` and yield a truncated read.
  - `createGrammarRegistry` is cached process-wide and invalidated by `vscode.extensions.onDidChange`. Don't rebuild per knit; the WASM init is expensive.
  - R-grammar resolution priority MUST flow through both `pickContribution` AND the `byScopeName` map. Two passes in `collectGrammarContributions` ensure `Registry.loadGrammar(scopeName)` resolves the same contribution that `scopeNameFor` would pick.
  - `GrammarRegistry.extractWithTheme` is the ONLY caller of vscode-textmate's `setTheme` / `tokenizeLine2`. It serializes concurrent extractions via an internal promise chain so two panels never see each other's theme between `setTheme` and the matching `tokenizeLine2` reads. The Knit highlighter path (`tokenizeLineForLanguage`) uses raw scope chains and is not affected by `setTheme`. Do not add a second `tokenizeLine2` caller without serializing through `extractWithTheme`.
  - The "Apply VS Code theme" toggle resolves a real palette from the active theme's TextMate `tokenColors` via `vscode-theme-palette.ts` and pushes it into the webview via a `__ravenVscodeThemePalette` postMessage. The webview's accept-regex `^(?:--raven-(?:bg|fg|c-[a-z]+): #(?:[0-9a-fA-F]{3,4}|[0-9a-fA-F]{6,8}); ?)+$` is the trust boundary between extension-host string assembly and `style.textContent` injection. It must stay in lockstep with `HEX_COLOR_RE` in `vscode-theme-palette.ts` (both accept exactly the 3/4/6/8-digit hex forms CSS supports). If you change the shape `paletteCssDeclarations` (in `knit-output.ts`) emits, update both regexes in the same commit or the toggle silently degrades to the GitHub palette.
  - `KnitOutputPanel.candidateThemeIds` + `vscode-theme-palette`'s `activeEditorBackground` disambiguation: the public VS Code API does NOT expose the active theme's id (`vscode.window.activeColorTheme` carries only `.kind`). When `window.autoDetectColorScheme` is true, VS Code swaps the active theme between `workbench.preferredLightColorTheme` and `workbench.preferredDarkColorTheme` based on the OS appearance. `activeColorTheme.kind` tells us the chosen theme's category (Light/Dark/HC), NOT the OS mode — so if both preferred-* themes happen to share a kind (a user who's configured both to dark themes is the canonical case), kind alone cannot pick the right one. The resolver therefore takes a list of candidate ids (ordered: kind-matching preferred-* first, then the other, then `workbench.colorTheme`) plus the webview's actual `--vscode-editor-background`, and picks the candidate whose theme JSON's `colors.editor.background` matches. The webview reports its bg in `knit-output.ts:reportThemeContext` (initial + on MutationObserver body-class change); the panel caches the latest value as `latestEditorBackground` and re-resolves on every update.
  - Live-font override mirrors the palette mechanism via `__ravenFontFamilies` postMessage. The webview's accept-regex `RAVEN_FONT_CSS_RE = /^(?:--raven-font-(?:text|mono): [^;{}<>\\\\\n\r\t\f\v\0]+; ?){2}$/` PLUS `fontCssIsComplete` (both names, no dups, no extras) PLUS `fontCssHasBareParens` (rejects `(`/`)` outside quoted family names — same rule the host's `hasBareParens` enforces) is the trust boundary between extension-host string assembly and `style.textContent` injection. All three checks must stay in lockstep with `fontsCssDeclarations` (output shape), `BANNED_CHAR_RE`, and `hasBareParens` (in `render-html.ts`). The override lives in a dedicated `<style id="raven-vscode-font-overrides">` element so it does NOT clobber the palette overlay's `<style id="raven-vscode-theme-overrides">` (the two channels set disjoint custom properties).
  - Theme-palette role resolution uses a small R-code corpus (`SAMPLE_CORPUS` in `vscode-theme-palette.ts`), not canonical probes per role. We tokenize each corpus line with both `tokenizeLineForLanguage` (raw scopes; theme-independent) and `tokenizeLine2ForLanguage` (binary tokens; theme-resolved colors), classify each token's role via the shared `scopeToRole` from `github-colors.ts`, then tally votes per role. Most-voted non-default color wins; ties go to first-seen. Single-canonical probes were too fragile across themes — `library(x)` only catches themes that style `support.function`, missing themes that target `entity.name.function` or rely on semantic-token-to-TextMate fallback through `meta.function-call`. The corpus is the answer: any theme that styles any of the scopes a representative R snippet exercises will be picked up. The vote-time `isWrongRoleForChain` filter prevents `"` (innermost-scope `punctuation.definition.string.*` but theme-colored via the outer `string.*`) from poisoning the punctuation role vote with the string color.
  - **YAML frontmatter is stripped in `renderKnitHtml` before invoking `markdown.api.render`**, so the rendered preview HTML never contains VS Code's `<table class="frontmatter">`. The strip happens in memory on the string passed to the injected `renderMarkdown` — the on-disk `.md` keeps its frontmatter and Pandoc HTML export / webview "Export ▾" both re-read it from disk (they depend on the YAML for title/author/output options/`pandoc_args`). `stripFrontmatter` and `extractFrontmatter` in `yaml-frontmatter.ts` share a private `findFrontmatterEnd` predicate; the lockstep test in `tests/bun/yaml-frontmatter.test.ts` guards against drift. The integration test in `editors/vscode/src/test/knit-preview-yaml-stripped.test.ts` pins the end-to-end behavior.
  - **Plot theming + SVG default for HTML targets**: `withSvgDevDefault` (in `output-options.ts`) injects `dev = 'svg'` into `chunkOpts` when `targetFormat === 'html'` and YAML didn't set one — both `runKnitCommand` (preview + editor-toolbar HTML export) and webview Export ▾'s reuse of the cached preview `.md` rely on this default. Non-HTML targets (PDF, Word) are untouched. The matching host-side substitution lives in `inlineLocalImagesAsDataUrls` (`inline-images.ts`): `<img src="figure/*.svg">` is replaced inline with sanitized `<svg class="raven-knit-plot">` markup (via `create_svg_sanitizer` + `tag_background_rects_with_document` from `plot/webview/`); user-included SVGs outside `figure/` keep the data-URL path. The CSS overlay in `knit-output.ts:applyTheme()` scopes recoloring to `svg.raven-knit-plot` and mirrors the Plot Viewer's selectors exactly. **jsdom is a runtime dependency** (lazily imported in `getSvgHostContext`; reset on `deactivate` via `disposeSvgHostContextForDeactivation`). The main `editors/vscode/tsconfig.json` sets `esModuleInterop: true` so the cross-module import from `src/knit/` into `src/plot/webview/sanitize.ts` (which default-imports the CJS `dompurify` package) type-checks under `module: commonjs`.

- **Plot viewer (Apply VS Code theme toggle + inline-SVG substrate)**
  - **Rendering substrate**: the webview renders plots as inline `<svg>` inside `<div class="plot-host">`, NOT as `<img src=httpgd-url>`. The substrate switch is load-bearing for the "Apply VS Code theme" toggle (parent CSS does not cascade into `<img>`-loaded SVG). Reverting to `<img>` breaks the toggle without warning. The new fetch path goes through `connect-src` (loopback HTTP); CSP `img-src` no longer carries `blob:` or `data:`.
  - **Sanitization invariant**: every SVG text from httpgd flows through `sanitize_svg` (in `editors/vscode/src/plot/webview/sanitize.ts`) before `{@html}`. DOMPurify is configured with `USE_PROFILES: { svg: true, svgFilters: true }`, `FORBID_TAGS: ['use', 'image', 'a', 'style', 'foreignObject', 'feImage']` (closes external-href, contextmenu-interception, CSS-`@import`, and arbitrary-HTML hosting attack surfaces), and `FORBID_ATTR: ['style']` (CSS-exfiltration defense behind the panel's `style-src 'unsafe-inline'`). The `class="httpgd"` and `viewBox` attributes are load-bearing for the overlay selectors; the regression test in `tests/bun/plot-webview-sanitize.test.ts` asserts they survive sanitization. DOMPurify is pinned in `editors/vscode/package.json` (NOT the root) — bumping the version requires re-running the sanitize tests.
  - **Toggle persistence + broadcast**: `PlotViewerPanel.THEME_PREFERENCE_KEY = 'raven.plot.applyVSCodeTheme'`. Every `set-theme-applied` message `await`s `globalState.update(...)` BEFORE dispatching `raven.plot.broadcastStateUpdate` so a concurrently-creating panel sees the new value via `readThemePreference` in its `build_html` seed. The orchestrator's `PlotServices.broadcastStateUpdate()` iterates open panels and calls `panel.postStateUpdate()`; the storage key string lives ONLY on `PlotViewerPanel.readThemePreference` (single source of truth).
  - **No-echo invariant**: the webview NEVER posts `set-theme-applied` in response to `state-update`. The toolbar button click is the only outbound source. The `SET_THEME_APPLIED` reducer no-op short-circuits when the value is unchanged (returns the same state reference so Svelte's `$state` skips the cascade). Tested at two layers: bun unit test of the reducer + VS Code integration test that drives a real WebviewPanel.
  - **onMount-ordering**: `App.svelte`'s `onMount` MUST install the `window.addEventListener('message', ...)` listener BEFORE posting `webview-ready`. The host responds synchronously with a `state-update`; a dropped initial state-update would leave the panel showing stale state.
  - **No host-initiated `state-update` from `create_panel`**: the host posts `state-update` ONLY from `on_webview_message`'s `webview-ready` branch and from `post_state_update()` callers (`notifyPlotAvailable`, `notifySessionEnded`, `broadcastStateUpdate`). Posting from inside `create_panel` would race the webview's listener install — the webview-ready round-trip is the single bottleneck that guarantees the listener is live before any state arrives.
  - **`bg=` parameter is `'#ffffff'`**: in the live preview fetch, `bg_for_fetch(themeApplied)` returns `'#ffffff'` regardless of the toggle. The CSS overlay hides httpgd's canvas rect when the toggle is on, and the webview body's `--vscode-editor-background` shows through. Re-introducing a theme-dependent bg value would require: (a) adding `state.themeApplied` to the fetch effect's dependencies (already there), (b) extending the SVG cache key to include `themeApplied` (since the rendered bytes would diverge), (c) updating the CSS overlay's hidden-rect assumption.
  - **SVG cache key shape**: `${sessionId}:${plotId}:${width}:${height}` (see `svg_cache_key` in `editors/vscode/src/plot/webview/state.ts`). sessionId-scoping prevents cross-session bleed; dimension-keying makes resize refetch (httpgd bakes width/height into the SVG at render time). The cache is FIFO over insertion order with a per-plot rule that drops other-size entries for the same `(sessionId, plotId)` on each insert — a resize gesture cannot evict other plots' history.
  - **Resize debounce**: `on_resize` in `App.svelte` debounces 100ms (trailing) before updating `dimensions`. Without this, a 1s drag at 60fps would produce ~60 fetch dispatches and blow the 50-entry cache. The debounce is load-bearing for smoke test #10 (post-quit history navigation survives a resize).
  - **Initial-render seed**: `build_html` writes `window.__ravenInitialPlotState = JSON.stringify({ themeApplied: <bool> })` so the Svelte `onMount` can dispatch the persisted value before first paint. The Svelte side deletes the global after reading so a `webview.html =` re-assignment doesn't replay a stale seed.

- **Rmd/Quarto semantic tokens**
  - `semantic_tokens_for_rmd_document` (in `crates/raven/src/handlers.rs`) walks chunks via `chunks::detect_chunks`, parses each R chunk body in isolation with `tree_sitter_r`, and rebases each token's line by `header_line + 1`. Lines are split on `\n` with trailing `\r` stripped so CRLF documents flow cleanly through the parser. Non-R chunk languages (`is_r_chunk_language` matches only `"r"` and `"rscript"`) are skipped — never widen this without explicit thought about what knitr engines actually evaluate as R.
  - The custom request `raven/semanticTokensForRString` exists for the knit-output webview pipeline to tokenize rendered code blocks without requiring the source document to be open as an LSP document. Keep its legend (single `function` entry) in sync with `textDocument/semanticTokens/full`.

- **Package-mode scope contribution**
  - `PackageScopeContribution`'s `r_internal_symbols`, `imported_symbols`, `test_helper_symbols`, and `test_attached_packages` are injected into the queried file's scope by `append_package_contribution` (recursive path, depth 0) and `ScopeStream::snapshot()` (streaming path). Both gate on `is_r_source_path(uri, root)`: files under `R/` see `r_internal_symbols` + `imported_symbols`; files under `tests/testthat/` see those PLUS `test_helper_symbols` (per-helper-path keyed so a helper does not self-inject) PLUS `test_attached_packages` (added to `scope.inherited_packages`, modelling `testthat::test_check`'s implicit `library(testthat)`). Files outside `R/` / `tests/testthat/` get NO contribution.
  - `ScopeStream::is_visible` and `symbol_for` must consult the pre-computed `contribution_symbol_names` after frame/prefix checks, otherwise out-of-scope diagnostics misattribute contribution-provided names to forward `source()` calls.
  - Test-only contributions (`test_helper_symbols`, `test_attached_packages`) MUST stay invisible to `R/` files. Mirror the gate by branching on `RFileKind::Test` inside the injection helpers, not by re-deriving the test-dir check from the path.

## Learnings

Items here have no natural code home. For invariants tied to a specific function, prefer a doc comment on that function — `git log` and existing module docs in `cross_file/scope.rs`, `cross_file/revalidation.rs`, `handlers.rs`, `package_library.rs`, `document_store.rs`, and `libpath_watcher.rs` already capture most subtle scope/diagnostics/package invariants.

### Cross-cutting code

- **tower-lsp 0.20** defaults to `buffer_unordered(4)` for all messages. Without `.concurrency_level(1)`, `did_change` notifications can be processed out of order, corrupting incremental text sync.
- **Module declarations**: when adding a top-level module to the `raven` crate, declare it in BOTH `crates/raven/src/lib.rs` AND `crates/raven/src/main.rs` — the lib and bin targets have separate module trees.
- **VS Code settings sync**: a new LSP-exposed setting must be wired in three places together — `editors/vscode/package.json` (schema), the shared init-options factory at `editors/vscode/src/initializationOptions.ts` (used by both `extension.ts` at runtime and `settings.test.ts` in tests), and the `SETTINGS_MAPPING` plus any named-value tests in `editors/vscode/src/test/settings.test.ts`. Don't reintroduce a separate runtime builder in `extension.ts` — it should keep delegating to `buildInitializationOptions`, so master defaults (e.g. `diagnostics.enabled`, cache forwarding) can't drift between runtime and tests. After editing `package.json`, regenerate the alphabetical settings index with `bun editors/vscode/scripts/generate-settings-reference.mjs`; the drift test `tests/bun/settings-reference.test.ts` gates this on CI.
- **`executeCommandProvider.commands`** must remain `vec![]` if the VS Code extension also registers commands manually via `vscode.commands.registerCommand` — `vscode-languageclient`'s `ExecuteCommandFeature` auto-registers each entry, causing "command already exists" crashes at startup. The server still handles `workspace/executeCommand` regardless.
- **VS Code language-scoped settings** like `[r]` editor overrides: read `config.inspect(...).globalValue` instead of merged `config.get(...)` before a global `config.update(...)`; otherwise workspace or folder overrides leak into global user settings.
- **File-type tracking** for non-R languages: track file type on the document itself, not solely from URI extensions — untitled buffers rely on `languageId`, and VS Code extension contributions should include mixed-case extensions when backend matching is case-insensitive.
- **Vendored R + R Markdown grammars**: Raven ships its own copies of `r.tmLanguage.json` and `rmd.tmLanguage.json` (from `REditorSupport/vscode-r-syntax`, MIT) under `editors/vscode/syntaxes/`, registered in `editors/vscode/package.json` as `contributes.grammars` for `source.r` / `text.html.markdown.rmarkdown`. **Why:** the knit-preview pipeline tokenizes on the workspace extension host, and the sibling grammar extensions (`vscode.r`, `REditorSupport.r-syntax`) have no `main` field so VS Code installs them UI-only — invisible to the workspace host in Remote SSH / Dev Container / WSL / Codespaces. Vendoring is the only path that makes the grammar bytes reachable from where the knit pipeline runs, and it doubles as the editor's `.Rmd` highlighting source in remote setups. **How to apply:** when touching anything that depends on the R or Rmd grammar being present (the install nag, the knit highlighter's priority list, docs that claim Raven defers all grammars to siblings), remember the touchpoints stay in lockstep — the manifest contribution (including the `embeddedLanguages` map for rmd, which must match the scopes the vendored grammar actually emits; `tests/bun/grammar-contribution-paths.test.ts` gates this on CI), both `R_GRAMMAR_PRIORITY` and `RMD_GRAMMAR_PRIORITY` (looked up via `GRAMMAR_PRIORITY_BY_LANGUAGE`) in `editors/vscode/src/knit/grammar-registry.ts` (Raven sits at the end of each as the self-resolving fallback), the vendored files plus `editors/vscode/syntaxes/SOURCE.md`, and the user-facing docs. The "Use VS Code's TextMate grammar instead of duplicate webview grammars" guidance still applies to the knit *webview* (don't extend Prism rules); shipping a vendored TextMate grammar via `contributes.grammars` is a different surface.

### Environment / tooling

- **macOS FSEvents** callbacks don't fire when the process is sandboxed by Claude Code; integration tests that depend on filesystem notifications require the sandbox to be disabled for the test runner (they pass fine in normal CI). Tests that genuinely cannot run under the sandbox (e.g. the 700K-row data-viewer scroll smoke tests, the knit iframe-load probe) self-skip via `isClaudeCodeSandbox()` in `editors/vscode/src/test/helper.ts` so a local run reports "skipped (sandbox)" rather than a flaky timeout failure.
- **VS Code suite cumulative state** can make `vscode.window.activeTextEditor` racy: `showTextDocument(doc)`'s promise resolves before VS Code finishes promoting the editor to active, and a focused webview can leave `activeTextEditor` `undefined`. Tests whose command handlers read `activeTextEditor` (e.g. the chunks suite's `run_chunk`) must call `awaitActive(editor)` after `showTextDocument` and before each `executeCommand`. See the helper in `editors/vscode/src/test/helper.ts` and the `execute_chunk_command` wrapper in `chunks.test.ts`.
- **Linux inotify**: libpath watching is recursive (one watch per descendant directory, ~10–20 per installed package). On old distros capped at the legacy 8192 default, warn users to raise `fs.inotify.max_user_watches`.
- **DOMPurify pin**: `dompurify@3.4.5` lives in `editors/vscode/package.json` dependencies (NOT the root). It's consumed by `editors/vscode/src/plot/webview/sanitize.ts` for the plot viewer's SVG sanitization and is exact-pinned (no `^`) because the DOMPurify SVG profile's behavior matters for the toggle to work. A version bump must re-pass `tests/bun/plot-webview-sanitize.test.ts` (which asserts `<svg class="httpgd">` and `viewBox` survive sanitization — load-bearing for the CSS overlay's `svg.httpgd > rect:first-of-type` selector). The package now ships its own types; do NOT add `@types/dompurify` (stub, npm warns it's unnecessary).

### Test harness

- **Bun test discovery** must NOT recurse into `editors/vscode/src/test` or `editors/vscode/out/test`; those are Mocha/`vscode-test` suites. Invoke them via a wrapper subprocess test from the root.
- **VS Code test compile** runs before `vscode-test`, so dependency type-resolution breakage in `node_modules` can make root `bun test` fail before any VS Code tests run.
- **`tests/bun/tsconfig.json`** exists only so VS Code's TS language server can resolve `bun:test` and bare imports like `fast-check` (whose `node_modules` lives under `editors/vscode/`) — bun itself runs the suite without it. The `include` list is deliberately narrow because several bun test files have long-standing latent type errors (incomplete VS Code mocks, discriminated-union access without narrowing, …). Add a file there only after `./editors/vscode/node_modules/.bin/tsc --project tests/bun/tsconfig.json --noEmit` stays clean.
