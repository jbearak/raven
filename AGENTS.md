When making significant changes:

1. Update the relevant user-facing docs under `docs/` (and `README.md` if appropriate) when user-visible behavior changes.
2. For changes to internal architecture, caching, performance, or debugging guidance, update `docs/development.md`.

5. Keep this file’s pointers, “Key invariants”, and “Learnings” sections accurate over time.

## Key invariants (do not regress)

- **Path resolution**
  - Backward directives (`@lsp-sourced-by`, `@lsp-run-by`, `@lsp-included-by`) must resolve relative to the file’s directory and **ignore `@lsp-cd`** (`PathContext::new`).
  - Forward directives (`@lsp-source`, `@lsp-run`, `@lsp-include`) and AST-detected `source()` calls must **respect `@lsp-cd`** when present (`PathContext::from_metadata`).
  - See `crates/raven/src/cross_file/path_resolve.rs` and `docs/cross-file.md`.

- **Workspace-root fallback**
  - Workspace-root fallback resolution is for **AST-detected `source()` calls only**, and only when there is **no explicit or inherited working directory**.
  - Never apply workspace-root fallback to backward directives.

- **LRU caches under locks**
  - Under read locks, use `peek()` (no promotion) to keep reads concurrent.
  - Under write locks, use `push()` (eviction/promotion happens there).

- **Diagnostics publishing**
  - Diagnostics must be published monotonically by document version.
  - Dependency-triggered revalidation may republish at the same version via an explicit “force republish” mechanism, but never older versions.

- **R subprocess safety**
  - Validate user-controlled inputs (e.g., package names) before using them.
  - Use timeouts for all subprocess calls.
  - Avoid interpolating user-controlled strings into R code.

## Learnings

- When building a new struct from `&T`, avoid `..*ref`/`..ref` in struct update syntax; clone the base (`..ref.clone()`) or construct fields explicitly.
- In tests that locate identifiers in generated code, avoid substring matches (e.g., `inner_func` inside `outer_func`). Prefer delimiter-aware search or node positions from the AST.
- Don’t recurse on identifier nodes just to “keep traversal going” — they have no children and the extra recursion can be removed for clarity.
- Avoid blocking filesystem I/O on LSP request threads; if a fallback check is needed, do it off-thread and revalidate via cache updates.
- Guard against deadlocks by avoiding nested async lock acquisition (especially around background indexer/state access).
- Use `saturating_add` (or equivalent) for sentinel end-of-line columns to prevent overflow.
- Don’t “paper over” missing files: file-existence checks must preserve accurate diagnostics instead of always returning `true`.
- Validate user-controlled package names/paths and skip suspicious values before using them in indexing or diagnostics.
- Avoid `expect()` in long-lived server paths (e.g., parser init); propagate errors with `Option`/`Result` instead.
- When adding new enum variants used in test match expressions, update all test helpers to keep matches exhaustive.
- Avoid hard-coded line/column positions in property tests; compute stable positions from the generated code or parsed nodes.
- When using `tokio::sync::watch`, guard waiters with in-flight state or revisions so late callers don’t hang on `changed()`.
- When using `u32::MAX` as an EOF sentinel, ensure function-scope filtering treats it as “outside any function” to avoid leaking locals.
- Distinguish “full EOF” (both line and column MAX) from end-of-line sentinels so scope filtering doesn’t drop function-local symbols at call sites.
- In proptest, prefer `prop_assume!` for invalid generated cases instead of returning early with non-unit values.
- Keep requirements docs aligned with the actual API surface (e.g., static construction vs insertion guarantees).
- Convert UTF-16 columns to byte offsets before constructing tree-sitter `Point`s; avoid mixing column units.
- When expanding point windows by a byte, advance to the next UTF-8 boundary to avoid mid-codepoint positions.
- Watch for O(n²) scope detection or duplicate-detection paths in large files; prefer indexed lookups when possible.
- Avoid duplicating local-scoping condition logic across functions; centralize to reduce drift.
- Be careful with hover/definition range calculations at line boundaries to avoid off-by-one bugs or invalid points.
- For removal events, use strict position comparisons (before, not at) to avoid removing symbols at their definition position.
- In hot scope-resolution paths, avoid repeated scans over large lists (e.g., function scopes per event); pre-compute mappings or cache lookups to prevent O(R·F) regressions.
- Keep doc comments and markdown examples aligned with current behavior (e.g., list= string literals support).
- Normalize markdown table spacing to match project lint expectations when adding spec tables.
- Avoid interpolating user-controlled strings into R code; pass help topics/packages as command args instead.
- R's `help()` function uses non-standard evaluation (NSE) for the `package` argument; wrap variables in parentheses to force evaluation: `help(topic, package = (pkg))` not `help(topic, package = pkg)`.
- Add language identifiers (e.g., `text`) to ASCII diagram/timeline fences to satisfy markdownlint (MD040).
- `tree_sitter::Tree` implements `Clone`; preserve ASTs in cloned index entries when reference searches depend on them.
- For intentionally-unused public APIs, either wire them into a caller or add a localized `#[allow(dead_code)]` with a brief comment to avoid warning noise.
- Base exports must be gated by `package_library_ready` to avoid using empty exports before R subprocess initialization completes.
- When adding parameters to scope resolution functions, update all callers including test helpers.
- For unannotated codebases (no @lsp-cd directives), use workspace-root fallback for source() path resolution - many R projects assume scripts run from the project root.
- The workspace-root fallback should ONLY apply when the file has no explicit @lsp-cd and no inherited working directory from parent chain; don't override intentional working directory configuration.
- When profiling LSP startup, use binary mode (not `text=True`) in Python subprocess calls - text mode can cause 30+ second delays due to buffering/encoding issues with LSP protocol.
- Profile with realistic workspaces: a workspace with many `library()` calls across files reveals bottlenecks that toy examples miss.
- In `did_open`, package exports are prefetched synchronously before scheduling diagnostics to avoid false-positive "unknown function" errors. In `did_change`, prefetching still happens in background with diagnostics waiting for it. Batch queries to minimize wait time.
- Filter already-cached packages before querying R to avoid redundant subprocess calls.
- 94% of CRAN packages use explicit `export()` directives (roxygen2); only ~6% use `exportPattern()`. Static NAMESPACE parsing eliminates R subprocess calls for most packages.
- The `base` package is special: it has no NAMESPACE file (only DESCRIPTION and INDEX). Handle this by checking for DESCRIPTION existence, not just NAMESPACE.
- INDEX files provide ~95% accuracy for packages using `exportPattern()` when R subprocess is unavailable - they list documented exports.
- Tiered loading (static → R → INDEX fallback) provides both speed and accuracy: sub-5ms for 94% of packages, accurate exports for all.
- When parsing structured R output (markers like `__PKG:name__`), handle missing end markers gracefully to avoid losing partial results.
- Always wrap R subprocess calls with `tokio::time::timeout()` to prevent hung R processes from blocking the LSP indefinitely.
- When iterating over identifiers in a file for diagnostics, cache scope resolution results by `(line, column)` — different columns on the same line can have different scopes (e.g., inside vs outside a single-line anonymous function like `sapply(x, function(p) p + 1)`). Line-only caching causes false positives for function parameters.
- Pre-compute a line-offset index (`Vec<usize>` of line start positions) to avoid repeated `text.lines().nth(row)` calls, which are O(n) each time.
- Use `HashSet` for membership checks in hot loops (package deduplication, workspace imports); `Vec::contains()` is O(n) and adds up quickly in completion/diagnostic paths.
- For background work queues, maintain a companion `HashSet` of pending URIs alongside the `VecDeque` to enable O(1) duplicate detection instead of O(n) `iter().any()` scans.
- Unbounded caches (`HashMap` with no eviction) are a memory leak in long-running LSP sessions. Always set a maximum size or use time-based expiry.
- When multiple AST traversals extract different kinds of symbols (assignments, S4 methods), merge them into a single recursive walk to avoid redundant tree iteration.
- Add early termination to workspace symbol collection (`break` when `max_results` reached) to avoid collecting thousands of symbols only to truncate them.
- Use `Arc<str>` for frequently-cloned string fields (like `ScopedSymbol.name`) and corresponding HashMap keys to reduce O(n) String allocations to O(1) atomic refcount increments.
- When converting a HashMap key type from `String` to `Arc<str>`, lookups via `.get()` and `.contains_key()` need `&str` arguments (e.g., `var.as_str()`) since `HashMap<Arc<str>, _>` implements `Borrow<str>`.
- `Arc<str>` does NOT have a stable `.as_str()` method (it's behind the `str_as_str` feature gate); use `&*arc` instead to get a `&str` reference.
- The `lru` crate's `LruCache::new()` requires `NonZeroUsize`; always clamp capacity with `.unwrap_or()` to a sensible default to avoid panics.
- `LruCache` doesn't derive `Debug`; implement `Debug` manually using `finish_non_exhaustive()` when wrapping in a struct.
- Use `peek()` (not `get()`) for LRU reads under `RwLock` read locks — `get()` promotes the entry (mutates) and requires `&mut self`, while `peek()` takes `&self` and keeps reads concurrent.
- When migrating from reject-at-limit to LRU eviction, update property tests: entries can be silently evicted, so "was_present" tracking based on prior inserts becomes unreliable.
- Tree-sitter wraps incomplete expressions inside blocks (e.g., `x <-` inside `if (TRUE) { }`) in a single ERROR node spanning the entire block. Use content-line detection to place diagnostics on the first line with actual code content (not structural keywords or braces). When *emitting* diagnostics, emit once per ERROR node and do not recurse into ERROR children to avoid duplicates (but content-line detection may need to traverse ERROR subtrees to find the first content token).
- Backward directives and working directory directives are header-only: they must appear before any code. When tracking the header boundary in the parsing loop, check *before* the `@lsp-` pre-filter so code lines without directives still end the header.
- VS Code auto-closing pairs (e.g., `(` → `()`) mean that when `onTypeFormatting` fires after Enter, the closing delimiter is already present in the document. A line containing only a closing delimiter (with optional whitespace) should be treated as "inside parens/braces" for indentation, not as a "closing delimiter line" — the delimiter was pushed down by the Enter press and the user wants content-level indentation.
- "Ambiguous parent" diagnostics are misleading: a file being sourced by multiple parents is a valid use case, not an error. The diagnostic was removed entirely (previously controlled by `ambiguous_parent_severity` config).
- Backward and forward directives support `line=eof` and `line=end` as synonyms for "use scope at end of file" — useful when a parent file sources multiple helpers and child functions need access to all of them.
- `detect_cycle` returns the closing edge (from a different file), not an edge from the queried file. Diagnostic positions from the closing edge are invalid for the queried file. Always use an outgoing edge from the diagnosed file for positioning.
- Dependency graph edge changes (adding/removing `source()` calls) must trigger revalidation of dependent files, not just interface hash changes. Otherwise cycle diagnostics on parent files become stale when a child's `source()` is commented out.
- The `visited` set in scope resolution must track the position at which each file was visited, not just whether it was visited. A file visited as a parent (backward edge) at a partial position must be re-visitable as a forward source() target at a wider position, otherwise symbols defined after the parent's call site are lost.
- For cooperative cancellation of synchronous diagnostic passes, use `DiagCancelToken` (wraps `Option<CancellationToken>`): `from_token()` in debounced paths, `never()` in tests/non-debounced. Check `is_cancelled()` between collector stages and every 64 iterations (`idx & 63 == 0`) in hot loops to balance responsiveness vs overhead.
- The edited file should use a shorter diagnostic debounce than dependent files for near-instant feedback. The cancellation token from `CrossFileRevalidationState::schedule()` must be cloned before the `tokio::select!` debounce block so it survives into the diagnostic computation phase.
- Diagnostic computation must not hold the `RwLock` read lock during expensive scope resolution. Build a `DiagnosticsSnapshot` under the lock (captures artifacts, metadata, graph clone, config), release the lock, then compute diagnostics from the snapshot. This prevents write starvation on `did_change`.
- Thread cancellation checks (`is_cancelled`) into recursive scope resolution (`scope_at_position_with_graph_recursive`) so cancelled diagnostic tasks abort graph traversal immediately instead of completing the full DFS.
- Store `ScopeArtifacts` as `Arc<ScopeArtifacts>` in `DocumentState`, both `IndexEntry` types, and `DiagnosticsSnapshot`. The scope resolution closures (`F: Fn(&Url) -> Option<Arc<ScopeArtifacts>>`) return `Arc` clones (O(1) refcount increment) instead of deep-cloning `HashMap`/`Vec`/`FunctionScopeTree`. The function body only borrows via `Deref` so no other changes needed.
- When recursively expanding package exports, don't dedupe merged dependency worklists with `Vec::contains()`. Preserve traversal order with a `Vec`, but track membership in a companion `HashSet` to avoid quadratic scans.
- tower-lsp 0.20 defaults to `buffer_unordered(4)` for all messages. Without `.concurrency_level(1)`, `did_change` notifications can be processed out of order, corrupting incremental text sync.
- When propagating function-scoped packages across `source()` edges, treat nested source-call scopes as descendants of outer package scopes; exact interval equality breaks nested local `source()` inheritance.
- When adding a new VS Code-exposed LSP setting, update all three extension layers together: `editors/vscode/package.json` schema contribution, `editors/vscode/src/extension.ts` initialization-options forwarding, and `editors/vscode/src/test/settings.test.ts` so docs and Settings UI do not drift from server support.
- Avoid duplicating initialization-options builders or types between the VS Code runtime and tests; share a single factory/module so settings like cache forwarding and master defaults (for example `diagnostics.enabled`) cannot drift.
- When adding non-R language support, track file type on the document itself instead of inferring only from URI extensions; untitled buffers rely on `languageId`, and VS Code extension contributions should include mixed-case extensions when backend matching is case-insensitive.
- Keep Bun test discovery scoped away from the VS Code extension test tree; `editors/vscode/src/test` and `editors/vscode/out/test` are Mocha/`vscode-test` suites, so root `bun test` should invoke them via a wrapper subprocess test instead of trying to execute those files directly.
- Keep the VS Code test harness resilient to third-party `.d.ts` churn: `editors/vscode` compiles extension tests before launching `vscode-test`, so dependency type-resolution breakage in `node_modules` can make root `bun test` fail before any VS Code tests run.
- When updating VS Code language-scoped editor settings like `[r]`, read `config.inspect(...).globalValue` instead of merged `config.get(...)` before a global `config.update(...)`; otherwise workspace or folder overrides can be copied into global user settings.
- For path completion UX in VS Code, accepting a directory completion does not reliably re-trigger suggestions from the LSP item command alone; the extension should also watch document changes and manually trigger suggest when a completion inserts a trailing `/` inside `source()` or directive path contexts.
- For nested path completions, keep the completion label as the basename for display, but set `filter_text` to the full inserted path (for example `scripts/helpers.R`) or VS Code can filter out valid child suggestions after the user types a directory prefix like `scripts/`.
- File-path go-to-definition and completion base-directory resolution for AST-detected `source()` calls must use the same workspace-root fallback rules as dependency-graph and scope resolution: fallback is allowed only when there is no explicit or inherited working directory.
- Keep file-path intellisense directive handling aligned with cross-file directive support: `@lsp-source`, `@lsp-run`, and `@lsp-include` should behave the same in the editor, while workspace-root fallback remains AST-`source()`-only.
- In property tests over generated identifiers, don't assert usage presence/absence by name alone when the generator can reuse the same identifier in multiple roles; assert against the specific occurrence's byte range or AST position instead.
- Don’t reuse the generic R assignment extractor for Stan document symbols: constrained declarations like `real<lower=0, upper=1> foo;` contain `lower=` / `upper=` clauses that look assignment-like to tree-sitter-r, but the outline should surface the declared Stan identifier (`foo`) instead.
- Stan files parsed through `tree-sitter-r` can collapse large regions into a top-level `ERROR`; Stan navigation features like find-references must not depend on AST `identifier` descendants being available under the cursor.
- JAGS/Stan structural blocks (`model`, `data`, `parameters`, etc.) must not share the same heading-level semantics as decorative comment sections. Keep blocks as fixed top-level outline containers (`section_level = 0`) so in-block headings nest under the block instead of becoming top-level siblings.
- JAGS/Stan outline section ranges must be clamped to the nearest enclosing container end (block, loop, or function), and brace-less loop ranges may need body-aware end handling because `tree-sitter-r` can under-estimate later-line loop bodies.
- `notify` v6's `EventHandler` is synchronous, so bridging to tokio requires a `std::sync::mpsc` plus `spawn_blocking` — don't try to use `tokio::sync::mpsc` directly inside `recommended_watcher`.
- Libpath watching must use `RecursiveMode::Recursive`. `NonRecursive` misses in-place package upgrades — `install.packages("pkg")` for an already-installed package rewrites files *inside* `<libpath>/pkg/` without changing the libpath's listing, so no events reach `touched_from_events` and cached exports stay stale. Recursive inotify attaches one watch per descendant directory (not per file), so ~10–20 watches per installed package; warn users running large CRAN snapshots on old distros capped at the legacy 8192 default to raise `fs.inotify.max_user_watches`.
- Skip `00LOCK-<pkg>` staging directories when snapshotting a libpath; they appear mid-install and would otherwise look like new packages that immediately disappear.
- When a filesystem-triggered revalidation runs, the document version has not changed — call `CrossFileDiagnosticsGate::mark_force_republish(uri)` before `publish_diagnostics` or the monotonic gate will suppress the republish.
- `PackageLibrary::invalidate_many` must also drop `combined_exports` entries for meta-packages (`tidyverse`, `tidymodels`) whose `attached_packages` intersect the invalidated names — otherwise stale aggregate exports survive per-child invalidation.
- When adding a top-level module to the `raven` crate, declare it in BOTH `crates/raven/src/lib.rs` AND `crates/raven/src/main.rs` — both the lib target and the bin target have their own module trees.
- macOS FSEvents callbacks don't fire when the process is sandboxed by Claude Code; integration tests that depend on filesystem notifications require the sandbox to be disabled for the test runner (they pass fine in normal CI).
- `raven.refreshPackages` must rebuild `PackageLibrary` (via `rebuild_package_library`) and restart the libpath watcher, not just `clear_cache`. `.libPaths()` is captured at init and goes stale when `renv` switches projects, `.Rprofile` is edited, or the user calls `.libPaths(new, ...)`; reusing the old snapshot defeats the whole point of a manual refresh.
- `LibpathEvent::Dropped` must (a) clear the package cache, (b) force-republish diagnostics for every open document, and (c) attempt one watcher restart (gated by `allow_recovery` so a persistent failure can't loop). Without (b), users are stuck with stale diagnostics derived from the now-empty cache until session restart.
- `refresh_packages_command_body` computes its `cleared` return as `before - after` against the **same** `PackageLibrary`, so it under-reports when `raven.refreshPackages` swaps in a freshly-rebuilt (empty) library — the body's internal `before` is then 0 regardless of how many entries the old library held. `execute_command` must capture `cached_count()` from the *pre-rebuild* library and compute the user-visible "cleared N" delta against that snapshot, otherwise users always see "Raven: refreshed 0 cache entries" even after evicting hundreds of stale entries.
- The libpath consumer's "which docs are affected" filter must NOT hold the `WorldState` read lock across `scope_at_position_with_graph` calls — same rule as diagnostic computation. Snapshot the inputs under the lock (open docs + line indices, the union dependency neighborhood, artifacts/metadata maps for that neighborhood, an `extract_subgraph` clone, workspace folder, max depth, backward-dependency mode), then drop the guard before iterating. Holding the lock across N cross-file scope resolutions starves `did_change` writers in workspaces with many open files.
- `PackageLibrary::invalidate_many` must transitively expand the dependent set: if A Depends B Depends C and C is invalidated, both B's and A's `combined_exports` must be dropped. A single-pass filter over immediate `depends`/`attached_packages` misses transitive dependents. Use a worklist algorithm that iterates until no new dependents are found.
- In `restart_libpath_watcher`, spawn the consumer (`run_libpath_consumer`) **before** calling `spawn_watcher`. If `spawn_watcher` sends `LibpathEvent::Dropped` and returns `None`, the consumer must already be listening on `rx` to execute the recovery path (clear cache, force-republish). Otherwise the `Dropped` event is lost and open documents keep stale diagnostics.
- Package prefetch paths (`refreshPackages`, `did_open`, `did_change`) must snapshot the full dependency neighborhood (via `collect_neighborhood` / `build_package_scope_snapshot`), not just open documents, to discover inherited packages from closed parent files in `source()` chains. Scope resolution puts forward-chain packages in `loaded_packages` (not `inherited_packages`), so collect both.
- After `rebuild_package_library` swaps in a fresh `PackageLibrary`, always call `prefetch_packages_for_open_documents` before publishing diagnostics — otherwise the empty cache causes transient false-positive "unknown function" diagnostics for package exports.
- When establishing a watcher baseline snapshot, capture it before `tokio::spawn` (not inside the spawned task) using `tokio::task::block_in_place`. Events queued in the notify channel between `watcher.watch()` registration and the task's first execution would otherwise be diffed against a baseline that already includes the change, silently producing a zero-delta and missing the cache invalidation.
- Do NOT list commands in `executeCommandProvider.commands` that the extension also registers manually via `vscode.commands.registerCommand`. `vscode-languageclient`'s `ExecuteCommandFeature` auto-registers VS Code commands for every entry in that list during `initializeFeatures`, causing "command already exists" crashes at startup. Keep `executeCommandProvider.commands` empty (`vec![]`) and let the extension own command registration; the server still handles `workspace/executeCommand` requests regardless.
- The "pending cache" suppression for "Undefined variable" diagnostics (`package_cache_pending`) must gate on `package_exists()` in addition to `!is_cached_sync()`. A not-installed package is also never cached, so `is_cached_sync` alone returns false for both "loading now" and "not installed at all" — without the existence check, uninstalled packages are permanently suppressed as if still loading. Cache the `package_exists()` results across the per-identifier loop (filesystem I/O, called once per unique package name).
- The "Package not installed" diagnostic gate must NOT use `r_subprocess.is_none()` alone. `ensure_package_library_initialized()` runs `RSubprocess::new()` synchronously on the async tokio thread — unlike `Backend::initialized()`, which uses `spawn_blocking` — so it can fail to discover R while `lib.initialize()` still populates `lib_paths` from `get_fallback_lib_paths()`. The state then has `package_library_ready = true` AND `r_subprocess = None`, and `Backend::initialized()` skips its overwrite (gate already true), permanently leaving `r_subprocess = None` in `state.package_library`. The correct gate is `lib_paths().is_empty() && r_subprocess().is_none()`: suppress only when there is no information at all. With non-empty `lib_paths`, `package_exists()` is filesystem-reliable regardless of whether an R subprocess is attached.
- `PackageLibrary::package_exists()` must consult the filesystem (and base packages) only — never the in-memory cache. `prefetch_packages()` queries R via `getNamespaceExports(asNamespace(pkg))` wrapped in `tryCatch(..., error = function(e) {})`, which silently swallows errors for not-installed packages. The R output emits `__PKG:name__` followed by zero export lines and `parse_multi_exports_output` records an empty-exports entry. `prefetch_packages()` then inserts that empty entry into the cache, so a never-installed package becomes "cached" with no exports. Trusting the cache in `package_exists()` causes it to return true for packages that aren't on disk, which permanently suppresses "Package 'X' is not installed" diagnostics after the first prefetch pass.
- The `BackwardDependencyMode::Auto` deferral in `collect_undefined_variables_*` is a function-level early-return: when `Auto && sourced_by.is_empty() && !workspace_scan_complete`, return immediately and emit no undefined-variable diagnostics. The post-scan force-republish (in `Backend::initialized`'s Task A) re-runs the collector with the full dep graph in place. An earlier per-identifier deferral let "package-related function calls" (function call after any prior `library()` in the file) emit immediately, so package issues like `library(lme4); lmer()` would surface before the scan completed. That heuristic was too coarse: it also let through false-positive diagnostics for `library(dplyr); helper_fn()` cases where `helper_fn` was actually defined in a sibling file not yet indexed. False flashes during cold start are worse than slightly delayed feedback for genuinely-broken package calls — defer everything until the dep graph is complete.
- The post-workspace-scan path in `Backend::initialized` must call `prefetch_packages_for_open_documents` BEFORE force-republishing open documents. `apply_workspace_index` builds the cross-file dependency graph, which makes auto-detected backward edges visible for the first time (e.g., `main.r` sources `data.r`, so `data.r`'s scope now inherits `main.r`'s `library()` calls — including transitive ones via `main.r`'s forward `source()` chain into `functions.r`). At `did_open`, those inherited packages weren't yet visible (workspace scan still running), so prefetch only saw the file's own `library()` calls. Without a post-scan prefetch, the now-visible inherited packages stay `is_cached_sync = false` forever (no later trigger caches them), and `package_exists()` returns true (they're installed), so `package_cache_pending = .any(...)` permanently silences `lmer()` after a local `library(lme4)` even though `lme4` itself is uninstalled — the symptom is "undefined-variable diagnostic for `lmer()` flashes briefly then disappears, while the bare `lmer` reference stays flagged" (the bare reference isn't a function call so doesn't hit the pending-suppression branch). Apply the same await-prefetch-before-republish pattern as `did_change_configuration`'s `package_settings_changed` branch.
- `state.workspace_imports` (populated from the workspace's NAMESPACE `importFrom()` entries) must store `(package, symbol)` pairs, not bare symbol names. The undefined-variable suppression that consults it must verify the source package is actually installed via `package_library.package_exists(pkg)` before suppressing. Otherwise: a workspace package whose NAMESPACE imports `lmer` from `lme4` will silence "Undefined variable: lmer" diagnostics in any file in the workspace, even when `lme4` is not installed — the import declaration is broken but the LSP behaves as if `lmer` were available. The bug manifests as a diagnostic that "appears for a moment" (cold-start publish before the workspace scan completes) "but then clears" (post-scan republish after `state.workspace_imports` has been populated). Apply the same gate in both `collect_undefined_variables_from_snapshot` and `collect_undefined_variables_position_aware`.
- Pairing `CrossFileDiagnosticsGate::can_publish` with `record_publish` on the same URI/version is a TOCTOU race under contention: `can_publish` takes only read locks on `last_published_version` and `force_republish`, so two concurrent same-version callers can both observe `force_active = true`, both pass the predicate, and then race into `record_publish` — both proceed off a single outstanding force marker, producing one extra publish per racing caller. - `ScopeEvent::Def` carries two distinct positions and they cannot be conflated. `line/column` is the *anchor* — it is the LHS identifier position, used by `annotate_event_function_scopes` to look up the containing function scope, and by any UI affordance that wants to point at the declaration site (it is also kept identical to `ScopedSymbol.defined_line/defined_column`, which drives hover and go-to-definition). `visible_from_line/visible_from_column` is the *effect position* — it is when the new binding actually enters scope, it is what `is_symbol_visible` (and the equivalent inline check in `scope_at_position_with_packages`) compares against, **and it is the key the timeline is sorted by** via `event_effect_position`. The two positions are *not* the same key: do not assume `line/column` orders the timeline. For `<-`/`=`/`<<-` (and `->`/`->>`) assignments whose value side is **not** a `function_definition` (after stripping `parenthesized_expression` wrappers via `unwrap_function_definition`), `visible_from` is the **end** of the whole assignment node — R evaluates the RHS *before* installing the LHS binding, so `merp <- merp` (and its multi-line variant `merp <-\n    merp`) must resolve the RHS in the enclosing scope and emit "Undefined variable: merp" when no outer `merp` exists. For assignments whose value side **is** a function definition (including paren-wrapped forms like `f <- (function() f())`), keep `visible_from = line/column` (the LHS); the body is delayed-evaluation, so recursive calls (`f <- function() f()`, mutually-recursive helpers, function-scoped `inner <- function() inner()`) must still resolve. The same RHS-before-binding rule applies to `assign("name", value)` calls — always use end-of-call as `visible_from` because R does not introspect the value argument as a function body. The symbol's own `defined_line/defined_column` (used for hover/go-to-definition) stays at the LHS, so navigation is unaffected. Regression coverage lives in `test_self_referential_assignment_rhs_is_undefined` (handlers.rs); the function-definition carve-out is exercised by `test_function_parameters_available_inside_function`, `test_nested_functions_separate_scopes`, and the property tests `prop_function_local_variable_scope_boundaries` / `prop_nested_function_separate_scopes` (which all query at positions inside function bodies with `hoist_globals=false` and rely on the LHS-position visibility for the enclosing function name).

- Cross-file diagnostic revalidation must walk THREE directions, not just backward dependents. (1) Backward (`get_transitive_dependents`): parents that source `uri` consume its exported interface.
